# 05 — Roadmap

*Last updated: 2026-05-13*

Three threads still open. (The basic "decouple rollout from training" overlap is now in via the off-policy prefetch feature in [`01_features.md`](01_features.md) — `FreshestPrefetchRolloutFunc` keeps vLLM generating while the trainer trains. What's listed below is the work that's *not* yet covered by that.)

## 1. Scaling to larger models and bigger batches

Currently we assume the policy and optimizer state fit on the trainer GPUs (2 by default), with a separate vLLM server on its own GPUs. That caps us at ~1-3B params full-FT or ~8B with LoRA on a single node. To go bigger:

**Single node (e.g. 4 GPUs).** Shard optimizer/gradients/params across trainer GPUs with [DeepSpeed ZeRO](https://www.deepspeed.ai/tutorials/zero/) (stage 2 shards optimizer+grads; stage 3 shards params too) or PyTorch FSDP. TRL supports both via an `accelerate config` file passed to `accelerate launch`. Combined with gradient checkpointing, this should let a 7-8B model train full-FT on 2 GPUs, or a 30B+ with LoRA. Note: vLLM's tensor parallelism (`tensor_parallel_size`) is independent — the inference server can shard a large model across its own GPUs without touching the trainer side. The NCCL weight-sync path in TRL already handles sharded trainers; the work is mostly configuration.

**Multi-node.** Same ZeRO-3 / FSDP story, but now the `accelerate` config specifies `num_machines > 1` and you coordinate ranks via `torchrun` or Slurm `srun`. The sharper question is where vLLM lives: one option is a single large vLLM server (tensor-parallel across one node) serving all trainer nodes, with NCCL weight sync over the cluster fabric; another is vLLM colocated per-node (TRL's `vllm_mode="colocate"`), which avoids the cross-node sync but currently doesn't play well with Inspect's async solver chain (see [`03_internals.md`](03_internals.md) "Where this could be simpler"). The right split depends on interconnect: with NVLink-class intra-node + slower inter-node, colocated vLLM is usually a win. A 4-node Layout B with DeepSpeed ZeRO-2 is wired up in [`examples/configs/4node/slurm.sbatch`](../examples/configs/4node/slurm.sbatch).

## 2. Inflight weight updates + variance-controlled off-policy (OlmoRL / VCPO)

Our prefetcher keeps vLLM busy generating but still **pauses** generation around the NCCL weight sync — the KV cache gets cleared on each sync. Olmo 3's RL recipe ([OlmoRL, Team Olmo 2025, arXiv:2512.13961](https://arxiv.org/abs/2512.13961)) goes further: push new weights into the vLLM server *without pausing generation or clearing the KV cache*, combined with **continuous batching** (dynamic prompt replacement so a finished rollout's slot is refilled immediately instead of waiting for the whole batch). Together these reportedly give ~4× throughput with no accuracy loss on a 32B model. [verl](https://verl.readthedocs.io/en/latest/advance/one_step_off.html) ships a ready-made "one step off" async trainer as a reference; TRL's [`trl.experimental.async_grpo`](https://github.com/huggingface/trl/blob/main/trl/experimental/async_grpo) gets you most of the way but still has the pause/resume hit.

Closing that last gap means either upstreaming a vLLM-side inflight-update endpoint or switching trainer (verl). Either way, the harder question for this repo is the variance side: at higher staleness, TRL's simple `min(ρ, 3.0)` clamp throws away information and a few outliers can dominate the gradient. VCPO ([arXiv:2602.17616](https://arxiv.org/abs/2602.17616)) addresses this with ESS-guided step scaling + an off-policy optimal baseline, reported stable up to k=128 steps off-policy. Pulling that into our loss would be a `GRPOTrainer._compute_loss` override (~200-300 LOC of math-heavy code).

## 3. Curriculum learning

The Inspect-owns-rollout design makes this unusually easy: since we already compose training data as Inspect `Task`s and scorers produce per-sample difficulty signals for free, curriculum here is mostly **dataset composition** — deciding which Inspect task and which subset of its samples to feed into the next batch.

**Two axes worth investigating:**

*Within-task difficulty scheduling.* Sort or re-weight samples within one task by a difficulty signal. Classic easy→hard ([E2H Reasoner, arXiv:2506.06632](https://arxiv.org/html/2506.06632)) works well when the dataset ships with a difficulty label (MATH has 1-5, GSM8K has grade level). More interesting is **variance-based** sampling — [VCRL, arXiv:2509.19803](https://arxiv.org/pdf/2509.19803) picks groups where the current policy has high reward variance (not all right, not all wrong), which is exactly where GRPO's advantage signal is non-degenerate. The `rewards/*/std` we already log per step is the hook; a sampler that preferentially draws from recent high-variance samples should drop in cleanly. [DUMP, arXiv:2504.09710](https://arxiv.org/abs/2504.09710) generalizes this to an automated, UCB-style distribution-level bandit — "pull the task whose expected learning progress is highest."

*Cross-task progression.* Your tool-calling → agentic math → research idea. Less well-studied for LLM RL than within-task difficulty, but conceptually straightforward: train on `math_agent` (single tool, short horizon) until the reward curve plateaus, then mix in GSM8K-style multi-step reasoning, then a research-style task with retrieval/browsing. The honest prerequisite is a stopping rule — you need to detect "we've learned this task" without overfitting on eval. Reward plateau + eval accuracy holding on a held-out Inspect task is the usual signal.

**What exists off the shelf.** [verl's multi-turn interaction system](https://verl.readthedocs.io/en/latest/sglang_multiturn/interaction_system.html) supports *named interactions* switchable at the sample level, which is the hook for agent/environment swaps mid-training — but the curriculum policy itself (when to advance, what to sample next) is still user code. TRL has no first-class curriculum support. The academic work ([PCuRL/VL-Cogito, arXiv:2507.22607](https://arxiv.org/html/2507.22607v1); [adaptive curriculum, arXiv:2504.05520](https://arxiv.org/html/2504.05520v1)) is mostly published as one-off training scripts rather than reusable libraries.

**Concretely for this repo.** A minimal curriculum hook would sit between the trainer and `inspect_eval`: instead of `task=<fixed Task>`, pass a `task_fn(step, recent_scores) -> Task` that can return different Inspect Tasks or different `dataset.filter(...)` slices over training. Because Inspect tasks compose (you can wrap any solver/scorer chain), this gives you both within-task and cross-task scheduling from a single hook. The tricky bit is state — the sampler needs a rolling buffer of per-sample reward statistics — but the per-step `.eval` logs already contain all of this; a small aggregator on top is ~100 lines. This feels like the most tractable of the three items listed here and the one most aligned with what the Inspect abstraction unlocks.
