# 004 Tool Calling Fixes — Session 1

Making `math_agent.py` (basic_agent + calculator tool) train end-to-end on Qwen 2.5. Three stacked problems, each surfaced by fixing the one above it.

## Problem 1: tools dropped by the provider

`TRLVLLMProvider.generate()` had `tools` and `tool_choice` marked `# noqa: ARG002`. The model never saw any tool schemas. Note: vLLM's `--enable-auto-tool-choice --tool-call-parser hermes` flags are **not** what's missing here — those configure vLLM's OpenAI-compatible server. `trl vllm-serve` exposes a different endpoint that returns raw text + token IDs; tool-call parsing has to happen client-side anyway.

### Fix

- Stash `tools` on `_QueueItem`, forward them in the payload.
- `_to_openai_tools()` wraps `ToolInfo` as `{"type": "function", "function": info.model_dump()}` — the form TRL's server forwards to `tokenizer.apply_chat_template(tools=...)`.
- Response parsing via `HFHandler.parse_assistant_response(text, tools)` from `inspect_ai.model._providers.util.hf_handler`. It dispatches on model-name substring, handles Qwen Hermes-XML `<tool_call>{...}</tool_call>`, Llama JSON, Mistral etc. Do **not** write your own regex — this handler already covers parse-error reporting, `parameters`→`arguments` rename, multi-call responses, and `shortuuid` call IDs.
- Batch homogeneity assert: one GRPO step = one task = one tool set. Never splits in practice.

Verified: eval-only run with the calculator reached 50% accuracy on 8 samples. Model actually invoked the tool.

## Problem 2: only the last turn got backpropped

`basic_agent` calls `generate()` once per turn. Each call overwrote `ModelOutput.metadata["trl_completion_data"]`, and the rollout read only `sample.output.metadata`. With `max_attempts=1`, the "last turn" is the `submit(answer=...)` call — the calculator-reasoning turns producing the actual reward signal were never in the gradient.

### Fix

Per-turn data lives on each `ChatMessageAssistant.metadata["trl_turn"]` now. `basic_agent` appends `state.output.message` to `state.messages` (`_basic_agent.py:189`), so the metadata rides along into the eval log. The rollout walks it:

```
completion_ids = C_0 ⧺ delta_{0→1} ⧺ C_1 ⧺ delta_{1→2} ⧺ … ⧺ C_N
env_mask       = [1]*|C_0| + [0]*|delta_{0→1}| + [1]*|C_1| + …
logprobs       = lp_0      + [0.0]*|delta|     + lp_1      + …
```

where `delta_{i→i+1}` is the tool-response tokens between turns. TRL's `GRPOTrainer` consumes `env_mask` as an extra field (`grpo_trainer.py:1749`), multiplied against the completion mask in the loss (`grpo_trainer.py:2052`). Environment tokens contribute zero gradient.

Single-turn fallback preserved so tldr/gsm8k keep working.

## Problem 3: BPE round-trip breaks token identity

The first attempt computed `delta_{i→i+1}` from the server's own `prompt_ids_{i+1}`:

```
delta_{i→i+1} = prompt_ids_{i+1}[len(prompt_ids_i) + len(completion_ids_i):]
```

with an invariant check `prompt_ids_{i+1}[:…] == prompt_ids_i + completion_ids_i`. The check fired mid-training:

```
first divergence at token 1667/1815.
expected = [4325, 3709, 2987,   296,  7934,   685]   # " someone getting either m" + "alar" + "ia"
got      = [4325, 3709, 2987, 68163,   476, 68163]   # " someone getting either malaria or malaria"
```

The model sampled "malaria" as `[296, 7934, 685]` = `[" m", "alar", "ia"]` (three suboptimal BPE pieces). On the next turn, the server re-tokenized the same decoded text and got `[68163]` = `[" malaria"]` — the canonical single token. **BPE encoding is not bijective over decode/encode round-trips.**

For GRPO this is fatal: the stored logprobs are for the tokens the policy actually sampled. If training-time recomputation sees a different token sequence for the same characters, the logprobs don't line up with the right positions in the sequence.

Root cause: TRL's `/chat/` endpoint applies the chat template server-side every turn, which re-tokenizes all prior content — including the model's own output. Any suboptimal BPE split in the rollout gets canonicalized away.

### Fix: client-side tokenization via `/generate/`

TRL vllm-serve's `/generate/` endpoint accepts `list[list[int]]` — pre-tokenized prompts. We construct `prompt_ids` ourselves:

- **First turn**: `tokenizer.apply_chat_template(msgs, tools=…, add_generation_prompt=True, tokenize=True, return_dict=False)` → initial prompt with tool schemas embedded.
- **Subsequent turns**: splice stored tokens + fresh delta.

```python
# rough sketch of _build_prompt_ids
before = apply_chat_template(msgs[:last_assistant+1], tools, add_generation_prompt=False, tokenize=False)
after  = apply_chat_template(msgs,                    tools, add_generation_prompt=True,  tokenize=False)
splice = before.rindex(tokenizer.eos_token) + len(tokenizer.eos_token)
delta_text = after[len(before[:splice]):]   # everything after the assistant's <|im_end|>
delta_ids  = tokenizer.encode(delta_text, add_special_tokens=False)
prompt_ids = prev_prompt_ids + prev_completion_ids + delta_ids
```

Why this works:
- `prev_completion_ids` ends exactly at `eos_token` (vLLM includes the stop token).
- `before` is the rendered conversation up through that same `<|im_end|>` plus whatever separator the template adds.
- Splicing at `rindex(eos_token) + len(eos_token)` lands us at the boundary between assistant content and environment content.
- `delta_text` contains only environment content (tool response wrapper, next turn's `<|im_start|>assistant\n`). No assistant text means no BPE round-trip risk.

Verified on Qwen 2.5 with Hermes tool calls: client-assembled `prompt_ids` matches the canonical `apply_chat_template` output token-for-token.

Generality: splice marker is `tokenizer.eos_token` — Qwen: `<|im_end|>`, Llama: `<|eot_id|>`. Works for any model whose template terminates assistant turns with its eos token (most do).

## Takeaway

Server-side chat templating is a leaky abstraction for multi-turn RL. Whenever the policy's exact token sequence matters across turns, tokenize client-side and own the prompt assembly.
